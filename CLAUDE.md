## Project Overview

**Monsoon Preparedness & Citizen Assistance** — A GenAI-powered PWA prototype for monsoon preparedness in Mumbai Suburban district. Built for a 2-hour hackathon with Next.js (App Router), Supabase/PostgreSQL, and API-based GenAI (OpenAI/Anthropic).

**Architecture**: Modular monolith PWA — single deployable unit (Next.js App Router), PWA-only (no native app), mock external data (IMD/CWC/ULB), API-based GenAI, phone OTP auth via Supabase, 12 Indian languages via GenAI translation.

## Architecture Overview

### Key Architectural Decisions (ADRs)

- **ADR 0001**: Modular monolith PWA — Next.js App Router, API routes as internal services, PWA-only, mock external data, API-based GenAI, phone OTP via Supabase, 12 languages via GenAI, offline-first critical paths, 1 district scope (Mumbai Suburban)
- **ADR 0002**: PostgreSQL schema for families, preparedness plans, checklists, weather alerts, localized alerts, shelters, evacuation routes, SOS requests, community reports, offline cache manifests, risk scores
- **ADR 0003**: GenAI prompt architecture for multilingual safety guidance (referenced but not in repo yet)

### Core Domain Model (from CONTEXT.md)

| Entity                 | Description                                                                          |
| ---------------------- | ------------------------------------------------------------------------------------ |
| **Citizen**            | User identified by phone (OTP-verified), belongs to Family                           |
| **Family**             | Group of Citizens sharing household; has location, housing type, vulnerability score |
| **Location**           | Admin (district/ward) + GPS + hazard zones                                           |
| **Preparedness Phase** | Pre-Monsoon (Apr–May), Active Monsoon (Jun–Sep), Post-Monsoon (Oct–Nov)              |
| **Preparedness Plan**  | Personalized per-family per-phase plan generated by GenAI                            |
| **Checklist Item**     | Atomic action in a plan: description, priority, phase, completion status             |
| **Weather Alert**      | Official IMD/CWC alert with type, severity, districts, validity                      |
| **Localized Alert**    | Weather alert translated + contextualized by GenAI for specific family               |
| **Shelter**            | Emergency shelter with capacity, facilities, occupancy                               |
| **Evacuation Route**   | Path from location to shelter with waypoints, conditions                             |
| **SOS Request**        | Emergency signal from citizen with location, type, status                            |
| **Community Report**   | Crowdsourced observation (waterlogging, road closure, etc.)                          |
| **Risk Score**         | Dynamic per-family score from weather, vulnerability, composition, preparedness      |

### Database Schema (ADR 0002)

Key tables: `profiles`, `families`, `family_members`, `preparedness_plans`, `checklist_items`, `weather_alerts`, `localized_alerts`, `shelters`, `evacuation_routes`, `sos_requests`, `community_reports`, `offline_cache_manifests`, `risk_score_history`

RLS policies enforce family-level data isolation. JSONB used for localized content and flexible metadata.

### Tech Stack

- **Framework**: Next.js 14+ (App Router)
- **Database**: Supabase (PostgreSQL + Auth + Realtime)
- **Auth**: Supabase Auth phone OTP
- **GenAI**: OpenAI/Anthropic API (plan generation, localization, Q&A, risk explanation)
- **i18n**: next-intl for static UI strings; GenAI for dynamic content (12 Indian languages)
- **PWA**: Service Worker for offline caching (checklists, routes, shelters, contacts)
- **Styling**: Tailwind CSS + shadcn/ui (configured)
- **Testing**: Jest + React Testing Library + @testing-library/jest-dom
- **Deployment**: Vercel (frontend) + Supabase (backend)

## Common Tasks

### Adding a new API route

Create under `/app/api/<route>/route.ts` using Next.js App Router conventions. Use Zod for input validation. Always use authenticated Supabase client with RLS.

### Working with Supabase

Use the Supabase client from `/lib/supabase.ts`. RLS policies enforce data isolation — always use the authenticated user's context. Server actions use `createServerSupabaseClient()`.

### Adding GenAI features

Use OpenAI/Anthropic SDK via `/lib/genai.ts`. System prompts handle 12-language translation. Keep prompts short for cost/latency. **Always validate GenAI output with Zod schemas** to prevent hallucination.

### Database changes

Manual SQL migrations via Supabase dashboard (hackathon constraint). Update TypeScript types in `/types` to match schema. Run `npm run db:types` to regenerate.

### Offline caching

Service Worker in `/public/sw.js` (to be created) caches personalized checklists, evacuation routes, shelters, emergency contacts per user device.

## HACKATHON DEPLOYMENT REQUIREMENTS — CRITICAL

### 1. FAST DEPLOYMENT

- **No build step blockers**: Ensure `npm run build` passes without errors
- **Environment variables**: Document all required env vars in `.env.example`
- **Vercel + Supabase**: One-click deploy to Vercel, Supabase project linked
- **PWA manifest**: `/public/manifest.json` with proper icons, shortcuts
- **Service Worker**: `/public/sw.js` for offline-first critical paths

### 2. TESTING (BASIC BUT MANDATORY)

- **Unit tests**: Jest + React Testing Library for all lib functions (genai, validators, utils, hooks)
- **API route tests**: Integration tests for `/api/family/*` endpoints
- **Run before deploy**: `npm run test:ci` must pass
- **Coverage target**: >70% on lib/ code (not required for hackathon but good practice)
- **Test file naming**: `*.test.ts` / `*.test.tsx` alongside source files

### 3. ACCESSIBILITY (WCAG 2.1 AA)

- **Semantic HTML**: Proper heading hierarchy, landmarks, form labels
- **ARIA**: Required roles, states, properties (aria-live for alerts, aria-expanded for accordions)
- **Keyboard navigation**: All interactive elements focusable, visible focus styles
- **Color contrast**: 4.5:1 minimum (Tailwind `text-gray-900` on `bg-white`, etc.)
- **Language**: `lang` attribute on `<html>`, `lang` on localized content
- **Screen readers**: Test with NVDA/VoiceOver; announce dynamic content
- **Reduced motion**: Respect `prefers-reduced-motion`
- **Touch targets**: Minimum 44x44px (Tailwind `min-h-[44px] min-w-[44px]`)

### 4. NO STATIC HARDCODED PAGES

- **All pages dynamic**: Use Server Components with `async` data fetching from Supabase/GenAI
- **No hardcoded HTML** for family data, alerts, shelters, plans
- **Dynamic routes**: `/family/[familyId]/plan`, `/family/[familyId]/alerts`, `/family/[familyId]/shelters`
- **Static generation only for**: landing page, auth pages, offline fallback

### 5. NO MOCK/FAKE DATA — REAL DATA ONLY

- **DELETE** `/lib/mock-data.ts` — all data from Supabase (seeded) or GenAI
- **Seed Supabase** with realistic Mumbai Suburban data via SQL migration
- **API routes** query Supabase directly with RLS
- **GenAI** generates plans/alerts from real family data + real weather alerts
- **Fallback**: If Supabase unavailable, show error state — never fake data

### 6. AI HALLUCINATION PREVENTION (CRITICAL FOR SAFETY)

- **System prompt**: Explicit "NEVER hallucinate evacuation routes, shelter locations, emergency numbers"
- **Grounding**: Every GenAI call includes real IMD/CWC/ULB data as context
- **Output validation**: Zod schemas for ALL GenAI responses (PlanGenerationOutput, AlertLocalizationOutput)
- **Citation requirement**: GenAI must cite source (IMD alert ID, NDMA guideline section)
- **Confidence scoring**: Return confidence score; if < 0.7, show "Verify with official source"
- **Fallback**: On validation failure, return safe default plan (not hallucinated)
- **Human-in-the-loop disclaimer**: "AI-generated guidance. Verify with local authorities."

### 7. SECURITY (HARDENED FOR PUBLIC DEPLOY)

- **RLS on ALL tables**: Family-level isolation enforced at DB level
- **API auth**: All API routes verify Supabase auth (phone OTP verified)
- **Input validation**: Zod schemas on every API route input
- **Rate limiting**: Vercel Edge middleware for API routes (100 req/min per IP)
- **CSP headers**: `next.config.js` headers with strict CSP
- **No secrets in client**: GenAI API keys only in server env
- **Sanitization**: DOMPurify on any user-generated content (community reports)
- **SOS/Reports**: Validate coordinates within district bounds

### 8. EFFICIENCY & PERFORMANCE

- **React Server Components**: Default for all pages; Client Components only for interactivity
- **Streaming**: `Suspense` boundaries for slow GenAI calls
- **Caching**: `next/cache` tags for family data, shelters; revalidate on updates
- **Bundle size**: Dynamic imports for heavy components (maps, charts)
- **Images**: `next/image` with Supabase storage, proper sizes
- **Database**: Indexes on `family_id`, `district`, `valid_from/valid_until`
- **GenAI**: Cache generated plans/alerts in Supabase (24hr TTL)

### 9. CODE QUALITY

- **TypeScript strict**: `strict: true`, no `any` (use `unknown` + type guards)
- **ESLint**: `eslint-config-next` + `plugin:@typescript-eslint/recommended`
- **Prettier**: Single quotes, trailing commas, 100 char line width
- **Husky pre-commit**: `lint-staged` runs prettier + eslint + typecheck
- **Zod schemas**: All API inputs, GenAI outputs, form data
- **Error boundaries**: React error boundaries on each page section
- **Logging**: Structured logging (pino) for server errors
